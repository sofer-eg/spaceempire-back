# Production — порт SP `TO_Production`

Спецификация автономного производства на станциях, портируется из старой
MySQL-процедуры `TO_Production` (`starwind/sql/db.sql:30608+`).

## Входы

- Каталог рецептов: `configs/balance.yaml`, секция `recipes` (см.
  `balance.Recipe`). Ключ — `station_type` (int), совпадает с
  `stations.type`.
- Состояние станций сектора (RAM `sectorState.statics.Stations`): для
  каждой станции читаются `Type`, `Built`, `InProgress`, `NextCycleAt`.
- Остатки в `cargo` (Postgres) по `owner_kind=Station, owner_id`.

## Выходы и побочные эффекты

- Списание входов (`recipe.inputs`) и зачисление выходов
  (`recipe.outputs`) на той же станции — `cargo` Postgres, в одной
  транзакции на цикл.
- Поля `stations.in_progress` и `stations.next_cycle_at` обновляются:
  - `false → true` + `next_cycle_at = now + cycle_time` — старт цикла.
  - `true → false` + `next_cycle_at = NULL` — завершение цикла.
- RAM-копия `sectorState.statics.Stations[i]` синхронизируется с
  Postgres.
- Метрика `production_cycles_total` (per-worker), инкрементируется на
  каждом успешно завершённом цикле.

## Алгоритм одного `Tick(ctx, state, now)`

Для каждой `station` в `state.statics.Stations`:

1. **Пропуск:** `!Built` или нет `recipe` для `station.Type` → ничего не
   делаем.
2. **Завершение цикла** (если `InProgress && NextCycleAt <= now`):
   1. Прочитать остатки входов из `cargo` (одной транзакцией).
   2. Если каждого входа в наличии хватает на `qty` — списать,
      добавить выходы, сбросить `InProgress=false`,
      `NextCycleAt=zero`, метрика `production_cycles_total++`.
   3. Если входов не хватает (race с торговлей или ручным переносом
      груза) — оставить `InProgress=true`, не сбрасывать
      `NextCycleAt`. На следующих тиках цикл «висит», пока ресурс не
      вернётся.
3. **Старт нового цикла** (если `!InProgress`):
   1. Прочитать остатки входов: для каждого `recipe.inputs[i]`
      требуется `qty` единиц.
   2. (опционально) прочитать остатки выходов: если у любого выхода
      `max > 0` и `current + qty > max`, цикл не стартует.
   3. Если все условия выполнены — `InProgress=true`,
      `NextCycleAt=now+cycle_time`. Списания входов **не** происходит
      — оно отложено до завершения цикла (повторяет оригинальное SP).

## Транзакционность

- Один `Tick` сектора может запустить/завершить циклы нескольких
  станций; каждый цикл — **отдельная транзакция** (insert/update в
  `cargo`, update `stations`). Это упрощает rollback: если у одной
  станции БД-ошибка, остальные не страдают.
- Чтение остатков `cargo` для проверки условий и последующая запись
  внутри одной транзакции — иначе условие проверяется на устаревших
  данных.

## Инварианты, сохранённые из SP

- Цикл стартует, только если **всех входов** хватает.
- Списание входов выполняется в момент **завершения** цикла, не старта.
  Игрок может в течение цикла «увидеть» вход и забрать его → цикл
  закончится без эффекта (см. шаг 2.3).
- `InProgress=true` запрещает повторный старт того же цикла.
- `NextCycleAt <= now` — единственное условие готовности к завершению,
  не считаем дробные/частичные выходы пропорционально `dt`.

## Намеренные упрощения относительно SP

- **Один `cycle_time` на рецепт.** В SP `cycle_time` хранится на каждой
  строке `station_goods_types`, итог — `sum`. Перенесём по
  необходимости, когда найдётся рецепт с разными временами.
- **Слиты `sgt.type=-1` (fuel) и `sgt.type=0` (input).** SP оставляет
  цикл активным, если `fuel` не хватает в момент завершения. У нас
  любой `input` ведёт себя одинаково — цикл «висит».
- **Не создаём предзаведённые строки `cargo` с quantity=0.** В SP это
  нужно для последующих JOIN'ов; в Go мы читаем `0`, если ряд
  отсутствует.
- **Acceptance-паритет** с SP — опционально (см. задача 3.6, критерии
  приёмки).

## Тесты

- Unit: один цикл — старт, ожидание тика > `NextCycleAt`, завершение.
- Unit: нет входов — старт пропускается.
- Unit: переполнение по выходу (`max`) — старт пропускается.
- Unit: вход исчез во время цикла — цикл «висит», метрика не растёт.
- Integration: 100 тиков, рецепт `Iron 5 → Microchips 3`, cycle=1s,
  начальный запас `Iron=1000` ⇒ через 100 тиков получены ожидаемые
  остатки.
